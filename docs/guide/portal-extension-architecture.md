# 门户扩展架构收敛方案

## 1. 这份文档解决什么问题

这份文档不讨论“能不能做”，只讨论“现在这套项目怎么做最优雅”。

当前项目已经有：

1. 主题 ZIP 导入与切换。
2. 插件安装、启停、卸载。
3. 门户运行时 `tabs / slots / widgets` 聚合接口。
4. Web Component 挂件最小闭环。

现在真正要收口的是边界。

如果边界不定死，后面就会继续出现下面这些破事：

1. 主题和插件职责混乱。
2. 业务模块到底算主题还是插件说不清。
3. 页面挂件到底该走插槽还是直接注入 HTML 没口径。
4. 后台管理面做着做着又混回一坨。

## 2. 总体原则

项目后续统一按“三层扩展模型”推进：

1. `theme`
   负责整站门户外观与页面骨架。
2. `module`
   负责独立业务能力与后台业务管理。
3. `widget`
   负责轻量页面增强，但需要继续拆成两种挂件模型。

这不是为了搞抽象，是为了防止每个扩展包都想接管一切。

## 3. 三层模型

### 3.1 主题层（Theme）

主题的职责只做三件事：

1. 定义访客看到的整站门户结构。
2. 提供页面骨架、视觉、导航、布局与插槽位置。
3. 暴露可挂载的 `slot` 给挂件或模块扩展消费。

主题不应该直接承担：

1. 独立业务数据库模型。
2. 后台 CRUD 管理逻辑。
3. 大量业务接口。
4. 模块级复杂业务流程。

主题的定位必须是：

`主题 = 整个门户的外壳与骨架`

### 3.2 模块层（Module）

模块负责独立业务能力。

典型例子：

1. 相册
2. 博客内容中心
3. 钢管产品库
4. 下载中心
5. 活动报名

模块可以包含：

1. 数据表 / 迁移脚本
2. 后端接口
3. 后台管理页面
4. 门户侧 tab / 页面块 / 列表 / 明细能力

模块不应该做的事：

1. 接管整个门户首页骨架
2. 替代主题
3. 直接篡改整站结构

模块的定位必须是：

`模块 = 能被主题容纳的独立业务能力`

### 3.3 挂件层（Widget）

挂件只负责轻量页面增强。

典型例子：

1. 每日一言
2. 侧边个人简介
3. 推荐文章卡片
4. 音乐播放器
5. live2d
6. 悬浮客服

但是挂件不能只有一个实现模型，必须拆成两类。

## 4. 两类挂件模型

### 4.1 `slot-widget`

这是主路线。

含义：

1. 主题页面先声明 `data-portal-slot=\"xxx\"`
2. 宿主运行时按 slot 收集挂件
3. 前端运行时把挂件渲染到对应 slot

推荐承载形式：

1. Web Component
2. 主题运行时动态加载 JS
3. 用 `tagName + configUrl + portalContext` 完成挂载

适用场景：

1. 首页右侧信息块
2. Hero 区域 banner
3. 推荐内容区
4. 商品卡片区
5. 相册预览区
6. 作者信息块

优点：

1. 有明确宿主契约。
2. 可视化编排容易做。
3. 排序、启停、权限、配置都能收口。
4. 适合 AI 生成标准化扩展。

缺点：

1. 要求主题作者预留 slot。
2. 协议比直接插 HTML 复杂一点。

### 4.2 `html-proxy-widget`

这是补充路线，不是主路线。

含义：

1. 挂件不走 slot。
2. 宿主在主题 HTML 输出前后做统一注入。
3. 插件声明把脚本 / 样式 / htmlSnippet 注入到 `head` 或 `body`。

它本质上更接近 PerfreeBlog 的 `HtmlRenderProxy` 思路。

适用场景：

1. live2d
2. 音乐播放器
3. 埋点脚本
4. 在线客服
5. 全站公告条
6. 悬浮按钮

优点：

1. 开发快。
2. 对主题侵入小。
3. 适合全站漂浮型、装饰型、脚本型增强。

缺点：

1. 结构化编排弱。
2. 多挂件并存时容易冲突。
3. 宿主很难知道它到底改了页面哪里。

所以这类挂件的定位必须明确：

`html-proxy-widget = 全局注入型挂件，不承担结构化页面编排`

## 5. 为什么不能只学 Perfree 那种注入方案

PerfreeBlog 的注入式挂件很适合博客 CMS，因为它解决的是：

1. 给现有页面塞点脚本
2. 加播放器
3. 加 live2d
4. 加浮层

但当前项目要解决的是更大的问题：

1. 主题切换
2. 模块扩展
3. 后台业务联动
4. 页面区域编排
5. AI 生成扩展包

如果把注入式挂件当主架构，会有这些问题：

1. 主题和挂件没有明确契约。
2. 多插件共存难管理。
3. 后台编排做不稳。
4. 业务块没法结构化扩展。

所以正确姿势不是二选一，而是：

1. 页面结构扩展走 `slot-widget`
2. 全局脚本装饰走 `html-proxy-widget`

## 6. 后台管理面的优雅拆法

### 6.1 主题中心

主题中心只管主题：

1. 上传主题 ZIP
2. 校验主题包
3. 预览主题
4. 切换主题
5. 卸载主题
6. 配置主题

主题中心不应该展示：

1. 模块
2. 页面挂件
3. 混合扩展

### 6.2 插件中心

插件中心只管非主题扩展：

1. `module`
2. `widget`
3. `hybrid`

插件中心负责：

1. 安装
2. 启停
3. 卸载
4. 类型过滤
5. 配置入口

### 6.3 编排中心

后续应该新增单独的“编排中心”。

它不等于插件中心，也不等于主题中心。

编排中心负责：

1. 当前主题有哪些 slot
2. 每个 slot 下挂了哪些 `slot-widget`
3. 顺序怎么排
4. 哪些启用，哪些停用
5. 哪些页面范围生效

`html-proxy-widget` 也可以挂到这里，但需要单独的分组与风险提示。

## 7. 开发者体验怎么做才不憨

### 7.1 主题开发者

主题开发者不应该拉整个宿主仓。

应该提供独立主题仓模板，至少包含：

1. `extension.yaml`
2. `theme.config.json`
3. `portal/` 构建产物输出规范
4. 本地预览脚本
5. 打包 ZIP 脚本

### 7.2 模块开发者

模块开发者可以需要后端与后台能力，但也应该尽量通过模板生成，不要手搓全链路。

### 7.3 挂件开发者

挂件开发者最简单，应该有两个模板：

1. `slot-widget-starter`
2. `html-proxy-widget-starter`

这样外部开发者不用先理解整个宿主内核。

## 8. 当前项目的正式收敛结论

后续统一按下面这套口径推进：

1. `theme = 门户外壳`
2. `module = 业务能力`
3. `widget = 页面增强`
4. `widget` 再拆成：
   `slot-widget`
   `html-proxy-widget`
5. 管理面拆成：
   `主题中心`
   `插件中心`
   `编排中心（后续新增）`

## 9. 推荐下一步顺序

别乱跳，按这个顺序推进：

1. 先把 `slot-widget / html-proxy-widget` 协议正式补到 Manifest 文档。
2. 再给后端 runtime 增加这两类挂件的统一运行态描述。
3. 再做“编排中心”后台页面，而不是继续堆主题预览花活。
4. 最后再抽独立主题仓与挂件仓模板。

## 10. 一句话结论

当前项目最优雅的收敛方式不是单一机制通吃，而是：

`主题管整站，模块管业务，挂件分结构化挂载与全局注入两条路线。`

这套边界清楚，后续最好扩，也最不容易越做越乱。
