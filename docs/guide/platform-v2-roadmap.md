# 宿主平台 V2 改造总控

## 1. 文档用途

这份文档不是宣传稿，也不是空谈架构的 PPT。

它只干 4 件事：

1. 记录宿主平台 V2 的目标和边界。
2. 约束后续改造顺序，避免边做边跑偏。
3. 记录当前进度、已确认决策、待办事项。
4. 作为后续代码改动、注释补充、联调验证的总入口。

## 2. 当前目标

现阶段要做的不是继续把现有项目硬堆成功能大杂烩，而是把现有工程逐步改造成一个可组合扩展平台。

平台核心目标：

1. 访客侧始终只有一个当前门户，默认通过 `/` 进入。
2. 后台可以切换当前门户主题，并管理扩展包启停。
3. 主题、业务模块、挂件都支持通过外部 ZIP 包导入。
4. 开发者可以独立仓库开发扩展，不必强依赖拉下整个主项目。
5. 宿主负责安装、注册、编排、启停、配置、回滚，不直接写死所有业务。
6. 为后续 AI 生成扩展包预留标准化入口。

## 3. 设计边界

### 3.1 本期必须支持

1. 主题包安装、启用、停用、切换。
2. 模块包安装、启用、停用、卸载。
3. 挂件包按页面区域进行编排挂载。
4. 门户路由、后台路由、后端接口的标准化注册。
5. 扩展包静态资源分发。
6. 基础配置管理、版本记录、失败回滚。

### 3.2 本期先不碰

1. 多站点同时运行。
2. SSR / 同构渲染。
3. 主题市场和远程下载中心。
4. 任意后端代码的完全热替换。
5. 复杂微前端编排平台。

别一上来就整宇宙平台，先把主链路打通，不然全是花活。

## 4. 对象模型

### 4.1 宿主内核（Kernel）

职责：

1. 扩展包安装、解压、校验、启停、卸载。
2. Manifest 解析与能力校验。
3. 路由、菜单、页面挂载面注册。
4. 配置、版本、发布、回滚记录。
5. 扩展生命周期管理。

### 4.2 扩展包（Extension Package）

统一使用扩展包模型，按职责分为 4 类：

1. `theme`
   当前门户主题，只允许同时激活一个。
2. `module`
   独立业务能力，例如相册、博客、钢管产品库。
3. `widget`
   小部件，例如每日一言、音乐播放器、横幅、浮窗。
4. `hybrid`
   同时包含门户、后台、接口、数据变更的组合型扩展。

### 4.3 挂载面（Surface）

门户侧：

1. `portal.route`
2. `portal.tab`
3. `portal.slot`
4. `portal.widget`

后台侧：

1. `admin.route`
2. `admin.menu`
3. `admin.page`
4. `admin.widget`

服务端：

1. `server.api`
2. `server.permission`
3. `server.migration`
4. `server.scheduler`

## 5. 技术原则

1. 优先复用现有插件安装链路、静态资源分发、后台动态路由能力。
2. 平台只接收构建后的产物，不直接接收 `.vue`、`.ts`、源码仓库。
3. 扩展开发采用独立仓库模式，开发态联调，发布态 ZIP 安装。
4. 访客侧互斥展示，运行时只能有一个生效门户主题。
5. 页面能力通过主题区域编排和挂件挂载扩展，不把所有页面都写死进主项目。
6. AI 后续只生成标准扩展包，不直接污染宿主主仓代码。

2. 已确认访客门户互斥，后台负责切换。
3. 已确认扩展包以 ZIP 导入，平台加载构建产物。

### 阶段 1：宿主内核抽象

状态：`未开始`

目标：

1. 梳理现有插件安装、启停、资源分发能力。
2. 抽出统一扩展注册中心和生命周期接口。
3. 定义扩展包元数据和运行态数据模型。

完成标准：

1. 宿主能识别扩展类型。
2. 宿主能记录启停状态、版本、安装信息。
3. 宿主能提供统一查询和状态变更接口。

### 阶段 2：主题运行时

状态：`进行中`

目标：

1. 建立当前门户主题注册和切换机制。
2. 让 `/` 始终渲染当前激活主题。
3. 支持后台主题切换、预览、回滚。

完成标准：

1. 同一时刻只能启用一个主题。
2. 切换主题不需要重新部署主项目。
3. 后台可见当前主题和历史切换记录。

### 阶段 3：模块与挂件编排

状态：`进行中`

目标：

1. 建立主题区域 `slot` 定义方式。
2. 支持模块和挂件向门户或后台挂载。
3. 支持典型样例，例如相册模块、每日一言挂件。

完成标准：

1. 新增模块不需要改主项目门户代码。
2. 挂件可以按页面区域启停和排序。
3. 后台能管理模块和挂件的安装与配置。

### 阶段 4：开发态与发布态分离

状态：`未开始`

目标：

1. 形成独立仓库开发扩展的标准方式。
2. 支持本地开发联调。
3. 支持发布构建、打包、ZIP 安装。

完成标准：

1. 扩展开发者不必改宿主仓源码。
2. 可以使用任意前端技术栈开发扩展，再产出构建结果。
3. 有明确的扩展目录规范、打包规范、联调规范。

### 阶段 5：AI 生成扩展包

状态：`未开始`

目标：

1. 固化 Manifest、配置 Schema、脚手架模板。
2. 让 AI 可以基于模板快速生成主题、模块、挂件骨架。
3. 为自动化生成后的人审、安装、调试建立流程。

完成标准：

1. AI 生成产物符合扩展包规范。
2. 生成内容能被宿主识别并安装。
3. 配置和挂载声明尽量 Schema 化。

## 7. 推荐实施顺序

严格按下面顺序推进，别乱跳：

1. 先补统一对象模型和 Manifest 草案。
2. 再做宿主内核注册中心和状态模型。
3. 再打通主题运行时切换。
4. 再做模块和挂件挂载编排。
5. 再补开发联调和 ZIP 发布链路。
6. 最后再考虑 AI 生成和模板生态。

## 8. 开发流程

每次开始一个子任务，统一按下面流程走：

1. 先在本文件更新“当前任务”和“预计影响范围”。
2. 阅读现有实现，确认是否已有相近能力。
3. 明确改动边界，只做最小必要修改。
4. 编码时补齐关键注释，尤其是新增结构、对外接口、核心流程。
5. 完成后更新本文件中的进度、结果、验证情况、遗留风险。

## 9. 注释规则

后续代码改动默认执行下面这套规则。

### 9.1 必须补注释的内容

1. 新增的类、接口、枚举、DTO、VO、实体、配置对象。
2. 新增的公共方法、扩展入口、生命周期方法。
3. Manifest 结构、配置字段、路由定义对象、挂载点对象。
4. 跨模块协作的关键流程代码。

### 9.2 注释写法要求

1. 注释必须解释业务含义、用途、约束，不写废话。
2. 字段注释优先说明“这个字段是干什么的”。
3. 方法注释优先说明“什么时候调用、影响什么、边界是什么”。
4. 复杂流程可以写少量块注释，但不要每行都写流水账。

### 9.3 不做的事

1. 不为了凑数给简单 getter/setter 写垃圾注释。
2. 不给一眼看懂的平铺代码硬塞废话。
3. 不一次性给全仓历史代码补注释，优先覆盖新增和改动区域。

## 11. 已确认决策

1. 门户访客入口使用 `/`，不在公开侧暴露主题前缀。
2. 访客只能看到一个当前生效门户，主题之间互斥。
3. 主题切换是后台行为，不是访客行为。
4. 扩展包必须支持外部 ZIP 导入。
5. 宿主导入的是构建产物，不是源码。
6. 扩展开发要允许独立仓库开发与联调。
